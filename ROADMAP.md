## 一、版本歷程

### v1.3.0-jhs（2026-05-20）— 💬 產生器接 LINE 即時通知（與國小端 v4.2.0 同步）

把既有審查模式的 LINE Flex 卡片通知機制延伸到「產生合規計畫」流程，與國小版 v4.2.0 同步上線。**原審查流程與產生器邏輯零異動**，只加 hook。

#### 新增通知點

| 階段 | LINE 卡片標題 | 帶的欄位 |
|---|---|---|
| started 🆕 | `[國中] 使用者送出 AI 產生計畫` | 學校 / 年級 / 領域 / 教科書版本 / 路線（快速/進階）/ 模型 |
| success ✅ | `[國中] AI 產生計畫完成` | 學校 / 年級 / 領域 / 議題 / 模型 / 生成字數 + 耗時 footer |
| failed ❌ | `[國中] AI 產生計畫失敗` | 學校 / 年級 / 領域 / 模型 / 錯誤訊息 + 耗時 footer |

#### 技術點

- 直接共用既有 `notifyAdmin()`（呼叫 `smes-e1dc3` Cloud Functions proxy）
- best-effort fire-and-forget + `keepalive:true`，失敗不擋使用者流程
- session ID（每分頁一份，sessionStorage）自動串連 started → success/failed，admin 在 LINE 一眼可看出是同一筆作業
- `app:'JHS'` + `[國中]` prefix 自動帶入，與國小版（`ES` / `[國小]`）明確分流

## 三-A、產生器後續路線圖（v1.2.0-jhs 之後）

> v1.2.0-jhs 上線後，「產生合規計畫」打開了一條全新的工具線（不只審查、還能反向產出）。
> 以下按 **ROI / 老師實際痛點** 排序，每條都標註「估計實作成本」與「能解掉的退件痛點」。

### 🔥 Tier 1：高 ROI 短期改良（1-3 天內可做）

#### G1-a. 5-2 進度表獨立 mini 產生器
- **痛點**：5-2 是國中老師最常被退件的項次（要寫滿 7/8 年級 20 週、9 年級 16-17 週、每週都要有內容、各領域分表，工程量極大）
- **做法**：在「產生合規計畫」內加一個切換 `📄 完整計畫` / `📅 只生 5-2 進度表`；後者只跑進度表，prompt 更專注、token 用量更省
- **加分**：教科書版本下拉直接連到康軒/翰林/南一各版本實際單元目錄資料庫
- **成本**：~1 天

#### G1-b. 「我去年的計畫」上傳改寫
- **痛點**：每年新學年度，老師都要把舊計畫翻出來改年份、改教科書版次、改教師名單
- **做法**：產生器加一個入口「📤 上傳去年計畫 PDF」→ AI 萃取結構 → 自動套今年度合規規則重寫
- **與審查端整合**：上傳的 PDF 直接用既有 PDF.js 擷取，不必新做
- **成本**：~1.5 天

#### G1-c. 議題融入節數計算器
- **痛點**：性別平等 6 節/學期、家庭教育 6 節/學年、生涯規劃 3 節/學期等法定議題節數，常常算錯被退件（連帶 3-2 / 6-3 都會跟著錯）
- **做法**：在 G2「議題融入」chip 旁顯示即時節數計算面板，超標 / 不足即標紅；產生時自動把節數寫進 prompt
- **成本**：~0.5 天

#### G1-d. 多領域批次產生
- **痛點**：一位老師往往要交多個領域（如 7 年級導師要做國/數/英/自/社全部）
- **做法**：G1 領域改成多選；每個領域跑一次 AI 呼叫；下載時打包 ZIP（或合併在一份 Word 內加目錄）
- **成本**：~1 天

### 🎯 Tier 2：中期能力擴充（1-2 週）

#### G2-a. 產 → 審 → 修 全自動閉環
- **痛點**：目前產完雖可一鍵送審，但審完還是要老師自己看建議手動回去改
- **做法**：審查結果若不通過 → AI 自動拿著「修正建議 + 原稿」生新版 → 再審 → 直到通過 or 達到 3 次重試上限
- **風險**：可能無限循環打 API，要設 retry cap
- **成本**：~3-5 天

#### G2-b. 即時規則違反檢測（編輯預覽時）
- **痛點**：老師手動編輯預覽內容時，可能不小心又寫成「歷史科」「【家庭教育】」等違規格式
- **做法**：contenteditable 加 input 監聽（debounce 1s）→ regex 比對違規模式 → 即時黃底標示 + tooltip 提示
- **規則表**：用既有 6 條鐵則自動推導出 regex（如 `/【.+教育】/g`）
- **成本**：~2 天

#### G2-c. 6 系列彈性學習產生器
- **痛點**：5 系列（部定）目前可生，但 6 系列（彈性學習）也有龐大文件需求（6-1 規劃表、6-2-1-1~3 統整探究、6-3 議題重點、6-4 進度表）
- **做法**：在產生器內加一個切換 `📚 部定領域` / `🎨 彈性學習課程`；後者 prompt 套 6 系列審查標準
- **特別注意**：6-2 國中分 7/8/9 三年級各一份（門數 3+3+5），prompt 邏輯需要動態套用
- **成本**：~5-7 天

#### G2-d. 跨學年自動降版
- **痛點**：116 學年度起，部分審查標準可能會調整
- **做法**：把「桃園市 115 國中規則」抽出成 JSON 設定檔，將來新增「116 規則」「117 規則」只要改設定不動程式
- **附加效果**：可同時維護 113-115 三年版本，方便老師查歷史
- **成本**：~3 天

### 🚀 Tier 3：跨工具與雲端（長期，1 個月+）

#### G3-a. 國中 / 國小 / 高中三端共用核心模組
- **狀態**：國小（v4.1.0）、國中（v1.2.0-jhs）已實作雙端產生器，下一步應該抽出 `core-generator.js` 共用，再透過設定檔差異化各端規則
- **做法**：兩 repo 都吃同一支 CDN 上的 `core-generator.js`，每端只維護自己的規則 JSON
- **未來高中端**：直接套 core + 寫高中規則設定即可，免再 fork
- **成本**：~5-7 天（要做整套 build pipeline）

#### G3-b. Google Sign-In + Firebase 雲端同步
- **痛點**：老師換電腦就要重設 API 金鑰、產生器表單預設值要重填
- **做法**：可選登入；登入後 API 金鑰、學校基本資料、最近產生的計畫都自動同步
- **隱私**：用 Firebase Authentication + Firestore，使用者資料只屬於自己
- **成本**：~7-10 天（含 Firebase 專案 + Authorized Domains + GitHub Actions Secrets）

#### G3-c. 校本範本庫
- **痛點**：同校老師常有共用的學校特色（如「鱻魚特色學園」食農、「閱讀策略融入」）但每位都要重打
- **做法**：學校建立一個共用範本（管理員權限），同校老師產生時可選「套用本校範本」自動帶入學校特色描述
- **與雲端同步整合**：共用 Firestore 文件
- **成本**：~5 天（依賴 G3-b）

#### G3-d. 教師授課節數 7-3 自動試算器
- **痛點**：7-3 全校教師授課節數表是教務處最頭痛的項次（每位老師授課節數、行政減課、導師減課要全部對齊「桃園市國民中學教師授課節數實施要點 114.09.12 修訂」）
- **做法**：獨立子工具：輸入「全校班級數 + 教師清單」→ 自動產出符合要點的節數表
- **成本**：~7-10 天

### 💡 Tier 4：UX 體驗加分（隨時可做）

| 想法 | 成本 | 痛點 |
|---|---|---|
| **範例體驗模式**：首次進產生模式自動預填一份示範資料，按下生成立刻看到結果 | 0.5 天 | 老師沒看到實際輸出前不敢用 |

| **匯出格式擴充**：HTML / Markdown / Google Docs / Notion | 各 0.5-2 天 | 學校要求格式不一 |

### 🩹 Tier 5：技術債務與健壯性

- [ ] **產生器 prompt 結構化測試**：寫一組「黃金測試」prompt（如「7 年級國語 + 康軒」），驗證 AI 輸出是否仍符合 6 道鐵則；若 Google 更新模型可早期偵測 regression
- [ ] **prompt 模板分離**：目前 prompt 寫死在 JS 字串內，未來抽出 `prompts/generator-jhs.md` 改用 fetch 載入，方便編輯
- [ ] **Token 用量監控**：產生 / 審查各跑一次大概消耗多少 token，給老師預估費用（雖然免費層多半夠用，但讓他們有概念）
- [ ] **Word 匯出排版優化**：目前匯出的 Word 表格在某些 Word 版本上欄寬會跑掉，可以加 colgroup 固定欄寬
- [ ] **無痕視窗 + PWA 不安裝場景的離線降級**：產生器需要 API，但 UI 框架應該離線也能瀏覽（顯示「無網路無法呼叫 AI」即可）
