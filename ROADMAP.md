### 🟢 容易做（立竿見影，建議下一波優先處理）

#### A. 🤖 無金鑰引導支援更多 AI 平台

- 分頁式引導：每次一個項次，完成後切換下一項
- **工作量估計**：1–2 天

#### S. 📸 掃描版 PDF 支援（OCR）
**問題**：許多學校的課程計畫是掃描版，目前完全無法擷取文字  
**方案一（免費）**：整合 Tesseract.js（開源前端 OCR），全程在瀏覽器執行  
**方案二（推薦）**：整合 Google Cloud Vision API，中文準確率極高（每月前 1,000 頁免費）  
**工作量估計**：1–2 天

#### T. 🗂️ 雙 PDF 比對（新舊版本差異分析）
**概念**：上傳「去年課程計畫」與「今年草稿」，AI 找出哪些段落可能沒有更新  
**技術**：同時上傳兩份 PDF，分別擷取文字，組裝比對 Prompt  
**工作量估計**：半天（UI 增加雙上傳區）

#### U. ☁️ Cloudflare Workers API Proxy（根本解決 CORS）
**問題**：OpenAI 在 GitHub Pages 環境下因 CORS 無法使用  
**方案**：部署 Cloudflare Workers Proxy，前端統一呼叫 Proxy，解鎖 Claude、Mistral 等更多模型  
**工作量估計**：半天（含 Cloudflare 設定）

#### V. 🌏 多縣市 / 跨學年度版本
**概念**：目前 Prompt 是「桃園市 115 學年度」專用  
**技術**：設計可抽換的「審查標準資料包」（`criteria/taoyuan-115.json`），工具首頁加縣市選擇器  
**目標**：打造全台最完整的課程計畫 AI 審查資料庫  
**工作量估計**：資料分離 1 天 + UI 選擇器 1 天

#### W. 🤖 多 AI 模型同時對比審查
**概念**：同一份 PDF + 同一個項次，同時送給 Gemini 2.5-flash 和 Gemini 2.5-pro，左右並排顯示  
**技術**：`Promise.allSettled()` 並行呼叫，介面左右兩欄  
**延伸**：「AI 共識模式」——兩個模型都說「表件通過」才顯示大綠旗  
**工作量估計**：半天 ~ 1 天

#### X. 🏆 審查品質評分（Scoring）
**概念**：目前 AI 只給「通過」或「待修正」，老師不知道嚴重程度  
**建議**：Prompt 要求 AI 給出結構化評分（0–100 分 + 必修缺失 / 建議修正分類）  
- 介面顯示「符合度 65%」進度條（綠 / 橘 / 紅漸變）
- 批次後可按分數由低到高排序
- **工作量估計**：1–2 天（Prompt 設計 + 結果解析）

#### Y. 🤝 Google 帳號登入 + 多裝置同步
**概念**：進度、提示詞模板都存 localStorage（只限一個瀏覽器），換電腦全部消失  
**技術**：整合 Google Identity API（OAuth 2.0），進度 / 模板 / 歷史同步到 Google Drive  
**工作量估計**：2–3 天

#### Z. 🏫 學校主任管理面板
**概念**：全校多位老師各自審查不同項次，主任需要一個總覽頁面  
**技術**：需要後端（Supabase）+ Google 帳號登入，支援任務指派 / 評論 / 全校彙整報告  
**注意**：這已是 SaaS 等級的產品  
**工作量估計**：1–2 週

#### LL. 🤖 AI 修改稿「全批次」自動生成
**概念**：II 功能是逐項按需生成；本功能是批次審查完成後，一鍵對所有待修正項目批次生成修正稿  
**流程**：
1. 批次審查完成 → 點「一鍵生成所有修正稿」
2. 逐項呼叫 API（帶入 PDF 原文 + 審查意見）
3. 所有修正稿匯入 Word 報告（各項次獨立區塊）
- **工作量估計**：1–2 天（含批次 API 管理 + Word 整合）

#### SS. 🔍 修正稿品質二次驗證（自動回審）
**概念**：II 生成修正稿後，自動把修正稿再送回 AI 驗證是否符合審查標準  
**流程**：生成修正稿 → 自動呼叫原審查 Prompt 驗證 → 顯示「✅ 修正稿已通過驗證」或「❌ 仍需修改」  
**效益**：讓老師確認修正稿真的可以通過，不需再手動重新上傳  
**工作量估計**：3–4 小時（在 II 基礎上延伸）

#### TT. 🌐 Google Sites / Classroom 整合
**概念**：將審查工具嵌入學校 Google Sites 課程計畫管理頁面，或整合到 Google Classroom  
**技術**：Google Sites 支援 iframe 嵌入、Classroom API 可讀取繳交檔案  
**工作量估計**：需要 Google Workspace 管理員權限，1–2 週

## 三、技術負債與需要注意的問題

| 問題 | 影響程度 | 目前狀態 | 建議處理方式 |
|------|----------|------|------|
| 單一 HTML 檔案持續增大（~145KB）| 🔴 高 | 已嚴重超過合理上限 | 拆分審查資料為獨立 `criteria.json`（fetch 載入），預計可縮減 40% |
| renderMarkdown 對複雜格式可能失準 | 🟡 中 | 已大幅改善但仍自製 | 考慮引入 marked.js（20KB，輕量）完全替換 |
| API 金鑰明文存 localStorage | 🟡 中 | 已有「金鑰不上傳」說明 | 加入 session 過期機制，提示定期清除 |
| Prompt 長度未截斷 | 🟡 中 | 有字數警告但未自動截斷 | 加入字數截斷（保留前 N 字 + 尾部 + 提示） |
| ~~批次審查無法跨頁儲存~~ | ✅ 已解決 | v3.4 PP 自動存檔每 5 分鐘 + 復原橫幅 | PP 已完成 |
| OpenAI CORS 問題（GitHub Pages） | 🟡 中 | 已有警告提示 | 完整解法需 API Proxy（對應建議 U） |
| ~~Service Worker 快取更新機制~~ | ✅ 已解決 | v3.3 版本化 CACHE_NAME + postMessage 通知 | HH 已完成 |
| 無自動化測試 | 🟢 低 | 全靠手動測試 | 加入基本 Playwright E2E 測試 |
| 無金鑰貼回結果無法驗證來源 | 🟢 低 | 老師自行把關 | 提示「請確認來源為正確的 AI 回覆」 |
| _fixTexts 記憶體未清空 | 🟢 低 | 批次結果存於記憶體，刷新即消失 | 搭配 PP 自動存檔一起處理 |
